## Docsの書き方

- 当面は日本語で書く
- いい切りの形で書く
- `cargo doc`したときに分かることは書かない
  - 「○○な関数である」
  - 「○○なモジュールである」
- 句読点をつけよう
- 可能な限り、型はリンクさせよう
- 動作例には`Example`をつける
- Unsafeな関数には`Safety`をつける

### 進捗度

- Coordinate型
- Ecef型
- Error型

## テスト

- Doc Testと普通のTestは両方書きたい
- 型に対応してテストを作成していく
- 同じファイルに書くと見通しが悪くなるので、
- testsの中にメゾットごとぐらいの粒度でファイルを作り、その中にファイルを作る
- テストが全て成功しないとPushできないようになっているので頑張る
- `cargo test`で実行できるテストは、軽量であること
- ランダムな要素を用いる場合は関数の初めに`random_`をつけること
- そもそも、ランダムなテストはよほどの必要性がない限り、非推奨
- ランダムなテストはproptestを用いて行うこと

## benchmark

- `cargo bench --features random`で回せるようになった

## ファイルの分離

- コードが複雑化する場合は機能ベースで分離していく
- 順序にも意味をちゃんと持たせる
- 公開されるメゾットは上に
- helperは下に
- 複数の機能からアクセスする可能性があるヘルパーはファイルに分離する
- ヘルパーはパフォーマンスと機能性を損ねない範囲で可能な限り抽象化する
