# `rustdoc`ガイドライン

Rustのコード内では `///` を用いて、ドキュメントを記述します。ドキュメントを維持するために以下のルールを適用する。。

## 1. 基本ルール

- **日本語・常体で記述**: 文末は「だ・である」調に統一。
- **句点**: 最後には必ず句点（`。`）をつける。ただし、箇条書きや単語の場合は不要。
- **テストを兼ねる**: 可能な限り、Docテストをつける。

## 2. 必須項目

- **関数の概要**: その関数や型が何をするものなのか。
- **パラメーター** (`# パラメーター`): 引数の説明。
- **バリデーション** (`# バリデーション`): ※存在する場合のみ。制約やパニック・エラーになる条件。
- **動作例** (`# 動作例`): コードブロックの直前には、必ず「タイトル:」をつけて何をしている例なのかを明記する。

### 例

````rust
/// 指定された値から全ての時間で有効な [`SingleId`] を構築する。
///
/// このコンストラクタは、与えられた `z`, `f`, `x`, `y` が各ズームレベルにおける範囲内にあるかを検証し、範囲外の場合は [`Error`] を返す。
///
/// # パラメーター
/// * `z` — ズームレベル（0–63の範囲が有効）
/// * `f` — Fインデックス（鉛直方向）
/// * `x` — Xインデックス（東西方向）
/// * `y` — Yインデックス（南北方向）
///
/// # バリデーション
/// - `z` が 63 を超える場合、[`Error::ZOutOfRange`] を返す。
/// - `f` がズームレベル `z` に対する `F_MIN[z]..=F_MAX[z]` の範囲外の場合、[`Error::FOutOfRange`] を返す。
/// - `x` または `y` が `0..=XY_MAX[z]` の範囲外の場合、それぞれ [`Error::XOutOfRange`]、[`Error::YOutOfRange`] を返す。
///
/// # 動作例
///
/// IDの作成:
/// ```
/// # use kasane_logic::SingleId;
/// let id = SingleId::new(5, 3, 2, 10).unwrap();
/// assert_eq!(id.to_string(), "5/3/2/10".to_string());
/// ```
///
/// 次元の範囲外の検知:
/// ```
/// # use kasane_logic::SingleId;
/// # use kasane_logic::Error;
/// let id = SingleId::new(3, 3, 2, 10);
/// assert_eq!(id, Err(Error::YOutOfRange{z:3, y:10}));
/// ```
///
/// ズームレベルの範囲外の検知:
/// ```
/// # use kasane_logic::SingleId;
/// # use kasane_logic::Error;
/// let id = SingleId::new(68, 3, 2, 10);
/// assert_eq!(id, Err(Error::ZOutOfRange { z:68 }));
/// ```
pub fn new(z: u8, f: i32, x: u32, y: u32) -> Result<SingleId, Error> {
    Self::new_with_temporal(z, f, x, y, TemporalId::whole())
}

````

## 3. ドキュメント記述の要否

### 書かなくて良いケース

- **自明な機能**: 関数名や型名を見ただけで、その役割や動作が完全に明らかな場合。

### 書くべきケース

- **関数名から発想される動作と異なるとき**: 利用者が直感的に想定する挙動と実際の処理にギャップがある場合。
- **エラーの発生理由がわかりにくいとき**: `Result` 型で処理される場合、なぜエラーになるのか、具体的にどの処理でエラーが発生するのか。
- **実行コストが不明瞭なとき**: `optimize_` など、関数名から計算コストが読み取れない場合は、実行コストについて注釈。「重い」「軽い」といった抽象的な表現は避け、「配列長に比例して計算量が増加する」「密度に依存する」といった**定量的・具体的な言葉**を使用する。

## 4. 設計思想や全体アーキテクチャについて

設計思想や、複数の型を横断するような大きな概念・思想に関する説明は`rustdoc`へ書くことは推奨しない。`/docs`にMarkdownファイルを作成して記述し、必要に応じてコード側のドキュメントからリンクを張る。
